IV.2 · DevOps and Site Reliability Engineering in depth

IV.1 showed you what cloud-native operations looks like; this topic takes you inside the engine room. We cover CI/CD pipelines at banking scale, Infrastructure as Code, platform engineering, incident command systems, the four DORA metrics in practice, and the human side of reliability. By the end you can lead a DevOps transformation or an SRE function in any tier-1 bank.

0. Primer — DevOps, SRE, and platform engineering Basic

Primer — read this even if you think you know it

DevOps is a cultural and technical movement (born 2009) that breaks down the wall between Development and Operations. SRE (Site Reliability Engineering, born at Google in 2003) is a specific implementation of DevOps: treat operations as software engineering, measure reliability with SLOs, and use error budgets to balance speed against stability. Platform engineering (the 2020s evolution) productises the internal platform so application teams can self-serve infrastructure without becoming infrastructure experts. In BFSI, all three are now mandatory for regulatory compliance, not optional optimisations.

0.1 IT-side anchor — your smartphone app updates

From what you already know — your laptop

Your smartphone updates apps automatically while you sleep. You do not call Apple or Samsung to ask when the next WhatsApp version will arrive; it just appears. Behind that is a pipeline: code commit → build → test → sign → deploy. That pipeline is CI/CD (Continuous Integration / Continuous Delivery). Now imagine the same thing for a bank’s loan origination system: a developer commits a fix for an interest-calculation bug, and within hours (not weeks) it is in production, tested, compliant, and observable. That requires the same discipline as your smartphone, but with far more safety checks because money is involved.

0.2 BFSI-side anchor — the monthly regulatory report

From what you already know — your bank account

Every month, your bank submits a regulatory report (Liquidity Coverage Ratio, NPA classification, FX exposure). The data comes from a dozen systems, stitched together by hand in spreadsheets, reviewed by three people, then uploaded to a portal. It takes four days and nobody trusts the numbers. DevOps thinking says: automate the extraction, automate the validation, automate the submission, and monitor the pipeline like a production service. SRE thinking adds: define an SLO (the report must be submitted by T-2 with 99.9% accuracy), measure error budget (how many rows were wrong last month), and hold someone accountable for reliability. Both mindsets turn a manual ordeal into a reliable system.

0.3 The five deep topics we will cover

  • CI/CD pipelines at scale — multi-environment, multi-region, with compliance gates and rollback strategies.
  • Infrastructure as Code (IaC) — Terraform, Pulumi, CDK, and the testing / security / drift-detection ecosystem around it.
  • Platform engineering — Internal Developer Platforms (IDP), golden paths, developer experience (DX), and the team topology that makes it work.
  • Incident management and reliability culture — incident command, blameless postmortems, toil budgets, chaos engineering at banking scale.
  • DORA metrics and continuous improvement — deploy frequency, lead time, MTTR, change failure rate; how to measure them honestly and improve them without gaming.

1. Why DevOps and SRE are now regulatory topics Basic

Why does DevOps matter enough to have its own regulatory expectations in BFSI?

  1. Speed without safety is now a compliance risk. EU DORA Article 8 (ICT risk management) and UK PRA SS2/13 both expect ‘rapid and reliable software updates’ as part of operational resilience. Slow patching is a reportable incident.
  2. Manual change is a concentration risk. Regulators now ask: ‘How many people know how to deploy the core ledger?’ If the answer is ‘two, and they are both on holiday,’ that is a risk. IaC and automated pipelines capture knowledge in code, not in heads.
  3. Auditability requires immutable history. GitOps (state declared in Git, reconciled automatically) gives regulators an auditable trail: who changed what, when, and what tests ran. Manual deployments leave no trail.
  4. Cloud spend optimisation depends on automation. FinOps (IV.1) requires tagging, rightsizing, and scheduling. All of those are implemented in CI/CD and IaC pipelines; without them, waste is guaranteed.
  5. Talent retention. Engineers leave banks that make them fight manual tickets to provision a VM. A modern IDP is a recruiting tool; its absence is a retention risk.

3. How CI/CD works at banking scale — phases, tools, environments, rollbacks Intermediate

3.1 CI/CD phases and gates

  • Source — commit to trunk or merge request; trigger webhook to CI server.
  • Build — compile, package, create container image. Deterministic: same source → same binary.
  • Unit test — fast, in-memory tests; fail the pipeline if coverage drops below threshold.
  • SAST (Static Application Security Testing) — SonarQube, Checkmarx, Semgrep, CodeQL; find SQL injection, hardcoded secrets, vulnerable dependencies.
  • SCA (Software Composition Analysis) — Snyk, JFrog Xray, Mend; check for CVEs in open-source libraries.
  • Artifact storage — signed container image, SBOM generated, provenance attestation (Sigstore/cosign).
  • DAST (Dynamic Application Security Testing) — OWASP ZAP, Burp Suite; test running application for runtime vulnerabilities.
  • Integration / contract tests — Pact, Spring Cloud Contract; verify API contracts with downstream systems.
  • Policy gate — OPA, Kyverno, Open Policy Agent; enforce ‘no critical CVEs,’ ‘resource limits declared,’ ‘no root user.’
  • Deploy strategy — canary (5% traffic), blue-green (instant switch), rolling (gradual replacement).
  • Smoke tests / synthetic monitoring — verify critical paths immediately after deploy.

3.3 Multi-environment promotion

BFSI requires at least four environments before production:

  • Dev — developer sandbox; unstable by design.
  • CI / Integration — shared testing of pull requests; ephemeral, spun up per branch.
  • Staging / Pre-prod — production-like, shared, sized for full integration tests; sometimes called ‘QA’ or ‘UAT’.
  • Production — live customer traffic; changes only via pipeline, never manual.

Promotion rules: artifact built once, promoted unchanged; configuration per environment via ConfigMap / external secrets; no ‘it works on my machine’ because containers are identical.

3.4 Rollback strategies

  • Blue-green — maintain two identical environments; switch traffic instantly; keep old version warm for instant rollback. Cost: 2× infrastructure.
  • Canary — route small percentage (5%, 10%, 25%) to new version; monitor error rate and latency; abort if SLOs breached; ramp to 100% over hours. Cost: minimal overhead.
  • Feature flags — code is deployed dark; flag enables feature per user cohort. Rollback is a flag flip, not a deploy. Tools: LaunchDarkly, Split, Unleash, OpenFeature (CNCF).
  • Database migrations — hardest part. Backward-compatible schema changes (expand-contract pattern); data migration as separate job; never drop column in same release that stops writing to it.

4. Infrastructure as Code — Terraform, GitOps, testing Intermediate

4.1 Infrastructure as Code — the basics

IaC means defining infrastructure (networks, VMs, databases, Kubernetes clusters, IAM policies) in declarative files, versioned in Git, applied automatically. Benefits: repeatability, auditability, drift detection, disaster recovery.

  • Terraform (HashiCorp, 2014) — de facto standard; HCL language; state file in S3 / Terraform Cloud; modules for reuse; providers for AWS, Azure, GCP, Kubernetes, and 2000+ others.
  • Pulumi (2018) — TypeScript, Python, Go, C# instead of HCL; imperative style; good for complex logic.
  • AWS CDK / CDKTF — define infrastructure in TypeScript/Python; synthesises to CloudFormation or Terraform.
  • CloudFormation / ARM / Deployment Manager — cloud-native IaC; vendor lock-in but tight integration.
  • Ansible / Chef / Puppet — configuration management (not strictly IaC); still used for VM bootstrapping.

4.2 Terraform at scale — patterns and pitfalls

  • Module registry — internal modules for ‘approved EKS cluster,’ ‘compliant S3 bucket with logging’; semantic versioning; enforced via CI.
  • Workspace per environment — dev, staging, prod Terraform workspaces; separate state files; separate IAM roles.
  • State file protection — state contains secrets; store in S3 with SSE-KMS, versioning, locking via DynamoDB.
  • Drift detection — nightly Terraform plan in CI; alert if reality differs from code (someone made a console change).
  • Policy as Code for IaC — Checkov, tfsec, Terrascan; enforce ‘no open security groups,’ ‘S3 must have versioning,’ ‘EKS must have private endpoints.’

4.3 GitOps — the next level of IaC

GitOps (coined by Weaveworks, 2017) applies the same Git-centric workflow to Kubernetes and infrastructure:

  1. Declarative: desired state in Git (YAML manifests, Helm charts, Terraform plans).
  2. Versioned and immutable: Git is the single source of truth.
  3. Pulled automatically: in-cluster controllers (Argo CD, Flux) continuously reconcile cluster to Git.
  4. Continuously reconciled: drift is auto-corrected; alerts fire if reconciliation fails.

Benefits for BFSI: audit trail (who merged what when); rollback (revert a Git commit); separation of duties (developers propose, approvers merge, controllers apply).

4.4 Testing infrastructure code

  • Static analysis — terraform validate, tflint, checkov.
  • Unit tests — Terratest (Go), kitchen-terraform (Ruby); spin up infra in test account, verify outputs, destroy.
  • Contract tests — validate that module outputs match what consuming services expect.
  • Integration tests — ephemeral environment spun up by CI, full application deployed, smoke tests run, environment destroyed.

5. Platform engineering — IDP, golden paths, team topologies Advanced

5.1 From DevOps to platform engineering

DevOps said: ‘you build it, you run it.’ Platform engineering says: ‘here is a paved road so you can run it without becoming an infrastructure expert.’ The platform team treats internal services as a product: APIs, documentation, SLAs, and a backlog driven by user research with application teams.

  • Internal Developer Platform (IDP) — a self-service layer hiding complexity: provision a cluster, deploy an app, rotate a secret, view logs, all via portal or API.
  • Golden paths — opinionated, supported templates: ‘Spring Boot on EKS with OTel, Argo CD, and Datadog.’ Teams can go off-road, but they own the support burden.
  • Developer Experience (DX) — measure: time from ‘I want a database’ to ‘I have a connection string’; time to first deploy for a new service; Mean Time To Patch (MTTP).

5.2 Team topologies for platform engineering

Reference: Team Topologies (Matthew Skelton, Manuel Pais, 2019) — the standard framework for organising engineering teams at scale.

5.3 IDP tooling landscape

  • Backstage (Spotify, CNCF Sandbox) — most adopted IDP framework; React frontend; plugin ecosystem; Software Catalog tracks ownership; Scaffolder creates services from templates; TechDocs renders docs. Users: JPMC, Standard Chartered, ING, Netflix.
  • Port — SaaS IDP; no-ops setup; strong workflows and scorecards.
  • Humanitec — Platform Orchestrator; drives Terraform/Helm/Argo; workflow-centric.
  • Cortex / Compass — SaaS IDPs with service catalog focus; Atlassian Compass integrates Jira/Confluence.

5.4 Service catalog and ownership

The IDP’s heart is a catalog: every service, who owns it, what it depends on, its SLOs, on-call rotation, documentation link, API specs. This is mandatory for regulatory reporting (DORA Article 8 requires mapping of critical dependencies).

6. Incident management, postmortems, chaos, and regulators Advanced

6.1 Incident management lifecycle

  1. Detection — alert from observability (SLO breach), customer report, or chaos test failure. Automated ticket creation; no ‘did you see my email?’
  2. Response — incident commander (IC) declared; communication channel created (Slack, Teams, PagerDuty); on-call engineers paged.
  3. Resolution — mitigate first (rollback, scale up, disable feature flag), then diagnose. MTTR measures mitigation time, not root-cause time.
  4. Recovery — confirm service restored; monitor for regression; close incident.
  5. Postmortem — within 24–48 hours; blameless; focus on system factors; action items tracked.

6.2 Incident Command System

Borrowed from emergency services and adopted by Google, Netflix, and major banks:

  • Incident Commander (IC) — coordinates, does not troubleshoot; frees engineers to fix.
  • Scribe — real-time timeline; every command run, every decision, with timestamp.
  • Communications Lead — internal updates to leadership, external to customers/regulators if needed.
  • Subject Matter Experts — the engineers actually fixing things; rotate out when tired.

6.3 Blameless postmortems

The gold standard: Google SRE book, Etsy, PagerDuty. Rules:

  1. Assume good faith; people did what made sense given the info.
  2. Focus on what made the system allow the mistake.
  3. Action items are system fixes, not training or ‘be more careful’.
  4. Share widely; psychological safety requires executive sponsorship.

6.4 Chaos engineering at banking scale

Principles from Netflix (Chaos Monkey, 2010) and formalised by Gremlin and AWS/Azure chaos services:

  • Start in dev/test — prove the tool, then graduate to staging, then controlled prod.
  • Minimize blast radius — one pod, one AZ, one microservice; abort conditions defined upfront.
  • Align to SLOs — the experiment validates that SLOs survive failure modes.
  • Regulatory relevance — UK PRA, MAS TRM, RBI all expect ‘scenario testing’; chaos engineering is the technical implementation.

7. DORA metrics, SLOs, error budgets, and toil Intermediate

7.1 The four DORA metrics

DORA (DevOps Research and Assessment, founded by Nicole Forsgren, Jez Humble, Gene Kim) publishes annual benchmarks based on 30,000+ professionals. The four key metrics:

MetricElite performerWhat it measures
Deployment FrequencyOn demand (multiple per day)How often we deliver value to production
Lead Time for Changes< 1 hour (commit to production)Velocity of the pipeline
Time to Restore Service (MTTR)< 1 hourResilience and recovery capability
Change Failure Rate< 5%Quality of the release process

These are outcome metrics, not vanity metrics like ‘lines of code.’ They correlate with organizational performance and, in BFSI, with operational resilience compliance.

7.2 Measuring honestly — avoiding gaming

  • Deployment frequency — count any change to production code or config; hotfixes count; rollbacks count.
  • Lead time — measure from commit (not story start) to production (not ‘ready for QA’).
  • MTTR — time from detection to mitigation (not root cause).
  • Change failure — any deployment causing incident, rollback, or hotfix; be honest, not political.

7.3 SLOs and error budgets in depth

SLI (Service Level Indicator) — quantitative measure: ‘latency of p99 request’ or ‘error rate.’

SLO (Service Level Objective) — target for the SLI: ‘p99 latency < 200ms over 28-day window.’

Error budget — (1 − SLO); how much ‘unreliability’ is allowed. If SLO is 99.9%, error budget is 0.1% = ~43 minutes/month.

Policy: when error budget is exhausted, freeze feature releases and invest in reliability. SRE team has veto power over launches if budget is spent.

7.4 Toil budget

Toil — repetitive, manual, automatable operational work. Google SRE rule: each SRE spends ≤ 50% of time on toil; ≥ 50% on engineering (automation, improving the platform). Measured quarterly; toil percentage is a team health metric.

8. Worked examples — five real-world DevOps/SRE journeys Intermediate

Worked example — Goldman Sachs — platform engineering for Marquee
  1. Marquee (Goldman’s digital platform for institutional clients) runs on an internal platform with strict golden paths; all services instrumented with proprietary observability stack.
  2. SRE function embedded in product teams; error budgets enforced by leadership.
  3. CI/CD deploys multiple times daily; DORA metrics tracked and published internally.
Worked example — JPMorgan Chase — SLOs and the Resilient Architecture Council
  1. JPMC’s Resilient Architecture Council defines SLOs for all Tier-1 services; services must meet SLOs to launch.
  2. Backstage-based IDP (internal) for service catalog and golden paths; GitOps via internal tools.
  3. Chaos testing program with AZ-level failure injection; results reviewed by risk committee.
Worked example — Standard Chartered — aXess platform DevOps transformation
  1. Platform engineering team created IDP on Azure with Backstage, Argo CD, and Datadog.
  2. Golden path: ‘Java/Node service on AKS with built-in SLO dashboards, canary deploys, and policy gates.’
  3. DORA metrics tracked: from quarterly deployments to multiple daily across critical services.
Worked example — Santander UK — retail bank CI/CD compliance pipeline
  1. Built mandatory compliance gates into CI/CD: every deploy must pass SAST, DAST, SCA, and policy checks.
  2. GitOps with Flux for Kubernetes workloads; Terraform for Azure infrastructure.
  3. Incident management integrated with FCA operational resilience reporting; automated timeline for postmortems.
Worked example — Hypothetical: 18-month DevOps transformation at a regional bank
  1. Month 1–2: Assessment — DORA baseline, toil survey, IDP prototype with Backstage.
  2. Month 3–6: Platform foundation — CI/CD on GitHub Actions, EKS clusters, Terraform modules, golden path templates.
  3. Month 7–12: Migration — pilot with 5 services; SLOs defined; SRE rotation established; error budget policy.
  4. Month 13–18: Scale — 50 services on platform; chaos testing program; FinOps integration; first regulator demonstration of automated recovery.

9. Questions a leader asks in any DevOps/SRE review Basic

  1. What are our four DORA metrics, and are we elite, high, medium, or low performers? What is the trend over the last 6 months?
  2. Which tier-1 services have defined SLOs and error budgets? How many error budgets were exhausted last quarter?
  3. What percentage of operational work is toil? What is the plan to reduce the highest-toil services?
  4. How long from ‘git commit’ to ‘production deploy’ for the median service? What is the bottleneck?
  5. What is our change failure rate? For failed changes, what is the MTTR? Do we prioritize speed or recovery?
  6. Is our IDP adoption voluntary or mandatory? What percentage of services use the golden path?
  7. How many manual steps remain in our production deployment process? Can any be automated?
  8. What is our incident MTTR trend? Are postmortems completed within 24 hours? Are action items resolved?
  9. Do we run chaos experiments in production? What was the last unexpected finding?
  10. What is our mean time to patch a critical CVE end-to-end? How does that compare to regulator expectations?
  11. Are SAST, DAST, and SCA gates mandatory in CI/CD? What exceptions exist, and who approved them?
  12. Do we have drift detection for IaC? When did we last find untracked manual changes in production?

10. Red flags + topic glossary Basic

Red flag — push back if you hear this
  • ‘We deploy 50 times a day, so we are elite.’ — Counting deploys without measuring change failure rate or MTTR is vanity. Elite means all four DORA metrics, not one.
  • ‘Our SLO is 99.99%’ — Without defining the SLI, the window, and the error budget policy, this is a slogan not an SLO.
  • ‘Toil is just part of the job.’ — No; toil >50% means the platform is failing and people will burn out.
  • ‘We don’t need an IDP; engineers can learn K8s.’ — They can, but they should focus on business logic. Platform teams exist to abstract complexity.
  • ‘GitOps is just CI/CD with Git.’ — No; GitOps is pull-based, continuously reconciled, and declarative. CI/CD alone lacks drift correction.
  • ‘We’ll automate later; speed first.’ — Manual process accretes and calcifies. Automate from day one; speed comes from automation.
  • ‘Terraform is too risky; console is safer.’ — Console changes are untracked, unaudited, and drift from code. IaC is safer with proper review and testing.
  • ‘Chaos engineering is just breaking things randomly.’ — Professional chaos is hypothesis-driven, blast-radius-limited, and aligned to SLOs.
  • ‘Postmortems should identify who made the mistake.’ — Blameful postmortems drive incidents underground. Blameless culture is safety-critical.
  • ‘Our MTTR is 2 hours because that is our SLA.’ — MTTR is measured from reality, not set by contract. If actual MTTR is longer, fix it; don’t redefine it.

Glossary for this topic

  • CI/CD — Continuous Integration / Continuous Delivery — automated build, test, and deploy pipeline.
  • SAST — Static Application Security Testing — analyse source code for vulnerabilities without executing it.
  • DAST — Dynamic Application Security Testing — test running application for vulnerabilities.
  • SCA — Software Composition Analysis — scan open-source dependencies for known CVEs.
  • IaC — Infrastructure as Code — define infrastructure in declarative files (Terraform, CloudFormation).
  • GitOps — Git-centric operations: desired state in Git, reconciled automatically by cluster controllers.
  • IDP — Internal Developer Platform — self-service layer abstracting infrastructure for application teams.
  • Golden path — Opinionated, supported template for common service types; teams can deviate but own the burden.
  • Platform engineering — Discipline of building internal platforms as products with UX and SLAs.
  • DORA metrics — Four metrics from DevOps Research and Assessment: deployment frequency, lead time, MTTR, change failure rate.
  • SLO — Service Level Objective — target reliability for a service over a time window (e.g., 99.9% availability).
  • SLI — Service Level Indicator — quantitative measure underlying an SLO (e.g., error rate, latency).
  • Error budget — (1 − SLO); allowed unreliability before feature releases pause.
  • Toil — Repetitive, manual, automatable operational work; SRE target is <50% of time.
  • MTTR — Mean Time To Restore — time from incident detection to mitigation.
  • MTTD — Mean Time To Detect — time from incident start to alert firing.
  • Chaos engineering — Discipline of injecting controlled failures to validate resilience.
  • Postmortem — Structured review of incident, blameless, action-item oriented.
  • Incident Commander — Role coordinating incident response, separate from technical fixers.
  • Drift detection — Automated comparison of deployed infrastructure against IaC definitions.
  • Canary deploy — Release to small percentage of traffic, monitor, then ramp up.
  • Blue-green — Two identical environments; instant traffic switch; instant rollback.
  • Feature flag — Toggle enabling code paths per user cohort; decouples deploy from release.
  • SBOM — Software Bill of Materials — inventory of components in an artifact; required by regulators.
  • Sigstore/cosign — Open-source tooling for signing and verifying container images.
